## 🔗 关系说明（重点！）

### 1对多关系（1:N）

图中有3条连线，都表示 **"1个设备对应多条数据"**：

```
Device（1个设备）
  ├── 1对多 ──> skin_sensor_data（多条皮肤数据）
  ├── 1对多 ──> environment_sensor_data（多条环境数据）
  └── 1对多 ──> device_status（多条状态记录）
```

**通俗解释**：
- 1台设备可以不断上报皮肤检测数据（每分钟一条）
- 1台设备可以不断上报环境检测数据
- 1台设备会定期更新状态信息

**举例**：
```
设备 DEV_001（安装在教室A）
  2026-04-24 10:00 → 皮肤数据：水分70%，油分35%，温度36.5°C
  2026-04-24 10:05 → 皮肤数据：水分68%，油分38%，温度36.8°C
  2026-04-24 10:10 → 皮肤数据：水分72%，油分32%，温度36.2°C
  ...（持续产生数据）
  
  2026-04-24 10:00 → 环境数据：温度25°C，湿度60%，PM2.5=35
  2026-04-24 10:05 → 环境数据：温度25.2°C，湿度58%，PM2.5=38
  ...（持续产生数据）
  
  2026-04-24 10:00 → 状态：电量85%，信号强
  2026-04-24 11:00 → 状态：电量80%，信号中
  ...（定期更新）
```

## 💡 关键术语解释

### PK（Primary Key）主键
**含义**：唯一标识一条记录的字段
**例子**：每个设备有唯一的 device_id，不会重复

### FK（Foreign Key）外键
**含义**：引用其他表主键的字段
**例子**：skin_sensor_data 表的 device_id 指向 Device 表的 device_id

### 1对多（1:N）
**含义**：一个主表记录对应多个子表记录
**例子**：1台设备 → 1000条传感器数据

### NOT NULL 非空
**含义**：这个字段必须有值，不能为空
**例子**：sensor_time（传感器时间）必须记录，不能缺失

### 自增（Auto Increment）
**含义**：每次插入数据时自动+1
**例子**：id字段：1, 2, 3, 4, 5...

## ❓ 老师可能问的问题

### Q1: 为什么用MongoDB而不是MySQL实现？
**A**:
```
传感器数据是JSON格式的半结构化数据，MongoDB更灵活。
而且MongoDB写入性能更好，实测比文件存储快34.9%。

ER图是按关系型数据库标准画的（PD设计），
但实际实现用的是MongoDB，更贴合业务场景。
```

### Q2: MongoDB中怎么实现外键？
**A**:
```
MongoDB是无模式数据库，不强制外键约束。
但我们通过应用程序逻辑保证数据完整性：
1. 写入时验证device_id存在
2. 查询时通过device_id关联查询
3. 删除设备时级联处理相关数据
```
