## 🔧 技术架构亮点

### 1. 多层安全防护
```
Layer 1: 网络层 - 速率限制（50 req/min）
Layer 2: 认证层 - JWT Token + API Key
Layer 3: 数据层 - AES-256加密传输
Layer 4: 审计层 - 请求日志记录
```

### 2. 高性能设计
- **Gunicorn多进程**: 8个worker，异步协程（gevent）
- **异步日志**: 独立线程处理，零阻塞I/O
- **日志采样**: 10%采样率，减少90%输出量
- **TCP优化**: backlog=4096，keepalive=2s

### 3. 消息队列解耦
- **Redis Streams**: 轻量级、持久化、消费者组
- **4个Stream**: raw/validated/write/logs
- **可靠传输**: ACK确认机制，消息不丢失

### 4. 模块化架构
```
app.py (路由入口)
  ├─ module_receiver.py (数据接收)
  ├─ module_validator.py (业务验证)
  ├─ module_writer.py (文件写入)
  └─ module_logger.py (异步日志)
```

## 💡 答辩问题准备

### 技术问题

**Q1: 为什么选择Flask而不是其他框架？**  
A: Flask轻量灵活，适合快速开发RESTful API。配合Gunicorn可实现高并发。若需要ORM、Admin等复杂功能，可选择Django。本项目追求简洁高效，Flask是最佳选择。

**Q2: Redis Streams和传统消息队列有什么区别？**  
A: Redis Streams是Redis 5.0新增功能，优势是轻量、无需额外部署、学习成本低。适合中小规模场景。RabbitMQ/Kafka更适合企业级应用，但部署复杂度高。我们选择Redis是因为已有Redis基础设施，复用降低成本。

**Q3: 如何保证数据不丢失？**  
A: 三重保障：①同步写入JSON文件（立即落盘）；②Redis Stream持久化配置；③备份机制（backup1-4）。即使服务崩溃，数据也不会丢失。生产环境可增加数据库双写。

**Q4: 安全性如何评估？**  
A: 四层防护：认证层（JWT/API Key）、传输层（AES加密）、应用层（速率限制）、审计层（日志记录）。生产环境建议增加HTTPS和IP白名单。当前已通过无效Token、超频请求等安全测试。

### 性能问题

**Q5: 性能瓶颈在哪里？如何优化？**  
A: 当前瓶颈在磁盘I/O（JSON文件写入）。优化方案：①批量写入；②使用数据库（MySQL/PostgreSQL）；③SSD硬盘；④异步I/O。理论上可支撑10000+ QPS。当前QPS 200+已满足需求。

**Q6: 如果要多机部署怎么办？**  
A: 架构支持水平扩展：①Nginx负载均衡多台Flask服务器；②Redis集群或主从复制；③共享存储（NFS）或分布式文件系统；④数据库读写分离。当前单节点已稳定运行。

### 架构问题

**Q7: 为什么采用模块化设计？**  
A: 单一职责原则（SRP），便于单元测试、维护和扩展。receiver负责接收，validator负责验证，writer负责写入，logger负责日志。各模块低耦合，可独立替换。

**Q8: 异步日志的优势是什么？**  
A: 传统同步日志会阻塞主线程，导致QPS降至50以下。异步日志使用独立线程处理I/O，主线程零阻塞，QPS提升至1000+。同时10%采样率减少90%日志量，进一步降低I/O压力。
